# 上下文窗口管理

处理超长文本、避免Token限制问题。

---

## 问题背景

```
大模型有上下文窗口限制：
- GPT-4: 128K tokens
- GPT-3.5: 16K tokens
- Claude-3: 200K tokens

超过限制会导致：
- 无法处理
- 回答截断
- 成本暴增
```

---

## 策略一：摘要压缩

```
适用于：长文档分析

原始输入（10000字）：
请分析这份长报告的核心观点。

优化后：
请分析以下报告的核心观点（已摘要至2000字）：
[摘要内容]

如果需要更详细的某部分，再单独请求。
```

---

## 策略二：分段处理

```
适用于：长篇文章、多文件

第一步：分段输入
"这是文章的第一部分（前5000字），请总结要点"

第二步：关键信息提取
"基于以上总结，提取关键数据和结论"

第三步：综合分析
"结合第一部分，总结全文核心观点"
```

---

## 策略三：渐进式QA

```
适用于：长代码库分析

第一步：
"这个文件（1000行）的主要功能是什么？"

第二步：
"基于你的理解，识别主要的类和函数"

第三步：
"详细分析 calculate() 函数的逻辑"

第四步：
"这段代码有什么潜在的安全问题？"
```

---

## 策略四：索引+召回

```
适用于：大量文档处理

第一步 - 建立索引：
"为以下10篇文档创建简短摘要索引：
doc1: [摘要]
doc2: [摘要]
..."

第二步 - 定向查询：
"关于'性能优化'，查看doc3和doc7的详细内容和代码示例"
```

---

## 实用技巧

| 技巧 | 说明 |
|------|------|
| 估算Token | 中文约2字符=1Token，英文约4字符=1Token |
| 预留空间 | 留20%buffer给回复 |
| 重要信息放前面 | 开头的信息权重更高 |
| 重要信息放末尾 | 结尾的信息也容易被记住 |

## Token估算

```
1000字中文 ≈ 500-600 tokens
1000字英文 ≈ 250-300 tokens

代码通常比自然语言消耗更多token
```
