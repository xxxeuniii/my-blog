# 元提示词 (Meta-Prompt)

任务拆解、调度与自我反思。

---

## 什么是元提示词

```
元提示词 = 提示词的提示词

核心思想：
- 让模型自己决定该怎么做
- 不是直接给步骤，而是让模型生成步骤
- 自我反思、自我纠正
- 复杂任务拆解调度
```

---

## 模式一：任务拆解调度

```
元提示词示例：

你是一个任务调度器。给定用户问题，请：
1. 分析问题需要哪些步骤
2. 把任务分解为子任务
3. 按顺序执行子任务
4. 组合结果给出最终答案

格式：
{
  "plan": ["步骤1", "步骤2", "步骤3"],
  "results": [...]
}

实际运行：
问题 → 生成计划 → 执行每一步 → 聚合结果
```

---

## 模式二：自我反思

```
元提示词示例：

请回答问题，然后进行自我反思：

步骤：
1. 给出初步答案
2. 批判自己：答案有没有问题？
3. 基于批判，给出改进后的最终答案

示例：

初步答案：...
批判：这里逻辑有问题，因为...
最终答案：修正后的答案...

好处：自动纠正错误
```

---

## 模式三：元优化（自动改进Prompt）

```
元提示词示例：

请分析这个Prompt，然后给出优化建议：

原始Prompt：[Prompt]
存在问题：[AI分析问题]
优化建议：[建议]
优化后的Prompt：[改进版本]

实际：可以让LLM自动优化自己用的Prompt！
```

## 完整的复杂任务处理流程

```
元提示词 + 反射 + 修正：

1. 理解：这个问题需要做什么？
2. 计划：分几步做？
3. 执行：一步步做
4. 检查：做错了吗？
5. 修正：如果错了，重新来
6. 输出：最终结果

用代码实现：
while not is_acceptable(answer):
    plan = generate_plan(question)
    answer = execute_plan(plan)
    critique = critique_answer(answer)
    if critique["is_good"]:
        break
```

## 自我反思Prompt示例

```
请回答数学题："3个苹果+2个苹果=？"

请按以下格式回答：

{
  "reasoning": "思考过程",
  "answer": "你的答案",
  "reflection": {
    "check": "检查一下这个答案对吗？",
    "correction": "如果错了，正确应该是？"
  }
}

第一次输出：
reasoning: 3+2=5
answer: 5
reflection:
  check: 对的，3个加2个是5个
  correction: 不需要

第二次（故意错）：
reasoning: 3+2=4
answer: 4
reflection:
  check: 错了！3+2=5，算错了
  correction: 5个
```

## 实际案例：代码审查

```
元提示词：
你是一个代码审查专家，请按以下流程审查：

1. 理解：这段代码想做什么？
2. 分析：有没有bug？性能问题？安全问题？
3. 自我验证：我的分析对吗？再仔细检查一遍
4. 建议：给出具体修复方案

代码：
[代码]

返回：
{
  "understanding": "理解",
  "issues": ["问题1", "问题2"],
  "self_check": "自我验证结果",
  "suggestions": ["修复建议"]
}
```

---

## 适用场景

| 场景 | 理由 |
|------|------|
| 复杂推理 | 需要分步验证 |
| 高要求任务 | 减少错误概率 |
| 自动优化 | 让模型改进自己 |
| 不确定性高 | 多步验证更可靠 |

---

## 权衡

```
优点：
- 更可靠，错误更少
- 输出质量更高
- 可观察性强（能看到思考过程）

缺点：
- 慢（多轮调用）
- 贵（Token消耗多）

建议：只在重要任务用
```
