# 数据库设计基础

## 一、数据库设计概述

### 1.1 设计原则

数据库设计的目标是创建一个**高效、可靠、可扩展**的数据存储方案。

**核心原则：**
- **数据完整性**：保证数据的准确性和一致性
- **性能优化**：支持高效的数据查询和更新
- **可扩展性**：能够适应未来业务需求的变化
- **安全性**：保护敏感数据

### 1.2 设计流程

```
需求分析 → 概念设计 → 逻辑设计 → 物理设计 → 实施部署
```

## 二、ER 图（实体关系图）

### 2.1 基本概念

| 概念 | 说明 | 示例 |
|------|------|------|
| **实体** | 现实世界中的对象 | 用户、订单、产品 |
| **属性** | 实体的特征 | 用户的姓名、年龄 |
| **关系** | 实体之间的联系 | 用户下订单 |
| **主键** | 唯一标识实体 | 用户ID |
| **外键** | 引用其他实体的主键 | 订单中的用户ID |

### 2.2 关系类型

| 关系类型 | 说明 | 示例 |
|----------|------|------|
| 一对一 (1:1) | 一个实体对应另一个实体 | 用户与用户档案 |
| 一对多 (1:N) | 一个实体对应多个实体 | 用户与订单 |
| 多对多 (M:N) | 多个实体相互对应 | 学生与课程 |

### 2.3 ER 图符号

```
矩形框：实体
椭圆形：属性
菱形框：关系
连接线：关联
```

## 三、数据库范式

### 3.1 第一范式 (1NF)

**规则**：每个属性都是原子的，不可再分。

```sql
-- 不符合 1NF（地址可再分）
CREATE TABLE users (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  address VARCHAR(200) -- 包含省、市、区
);

-- 符合 1NF
CREATE TABLE users (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  province VARCHAR(50),
  city VARCHAR(50),
  district VARCHAR(50)
);
```

### 3.2 第二范式 (2NF)

**规则**：满足 1NF，且非主键属性完全依赖于主键。

```sql
-- 不符合 2NF（课程名称部分依赖于课程ID）
CREATE TABLE student_courses (
  student_id INT,
  course_id INT,
  student_name VARCHAR(100),
  course_name VARCHAR(100),
  PRIMARY KEY (student_id, course_id)
);

-- 符合 2NF
CREATE TABLE student_courses (
  student_id INT,
  course_id INT,
  PRIMARY KEY (student_id, course_id)
);

CREATE TABLE courses (
  course_id INT PRIMARY KEY,
  course_name VARCHAR(100)
);
```

### 3.3 第三范式 (3NF)

**规则**：满足 2NF，且非主键属性不传递依赖于主键。

```sql
-- 不符合 3NF（部门名称传递依赖于部门ID）
CREATE TABLE employees (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  department_id INT,
  department_name VARCHAR(100)
);

-- 符合 3NF
CREATE TABLE employees (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  department_id INT,
  FOREIGN KEY (department_id) REFERENCES departments(id)
);

CREATE TABLE departments (
  id INT PRIMARY KEY,
  name VARCHAR(100)
);
```

### 3.4 范式对比

| 范式 | 规则 | 目的 |
|------|------|------|
| 1NF | 属性不可再分 | 消除重复组 |
| 2NF | 非主键完全依赖主键 | 消除部分依赖 |
| 3NF | 非主键不传递依赖 | 消除传递依赖 |

## 四、表设计实践

### 4.1 表结构设计

```sql
-- 用户表
CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID',
  username VARCHAR(50) NOT NULL UNIQUE COMMENT '用户名',
  email VARCHAR(100) NOT NULL UNIQUE COMMENT '邮箱',
  password_hash VARCHAR(255) NOT NULL COMMENT '密码哈希',
  phone VARCHAR(20) COMMENT '手机号',
  status ENUM('active', 'inactive', 'deleted') DEFAULT 'active' COMMENT '状态',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  INDEX idx_status (status),
  INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

### 4.2 外键约束

```sql
-- 订单表（含外键）
CREATE TABLE orders (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT NOT NULL,
  total_amount DECIMAL(10, 2) NOT NULL,
  status ENUM('pending', 'paid', 'shipped', 'completed') DEFAULT 'pending',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  
  -- 外键约束
  FOREIGN KEY (user_id) 
    REFERENCES users(id) 
    ON DELETE RESTRICT 
    ON UPDATE CASCADE
);
```

### 4.3 中间表（多对多关系）

```sql
-- 用户角色中间表
CREATE TABLE user_roles (
  user_id INT NOT NULL,
  role_id INT NOT NULL,
  PRIMARY KEY (user_id, role_id),
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  FOREIGN KEY (role_id) REFERENCES roles(id) ON DELETE CASCADE
);
```

## 五、数据类型选择

### 5.1 数值类型

| 类型 | 范围 | 适用场景 |
|------|------|----------|
| TINYINT | -128 ~ 127 | 状态标志 |
| SMALLINT | -32768 ~ 32767 | 小范围数值 |
| INT | -21亿 ~ 21亿 | 常规数值 |
| BIGINT | -9e18 ~ 9e18 | 大数值 |
| DECIMAL | 精确小数 | 金额、数量 |
| FLOAT/DOUBLE | 近似小数 | 科学计算 |

### 5.2 字符串类型

| 类型 | 特点 | 适用场景 |
|------|------|----------|
| CHAR(n) | 固定长度 | 短字符串、编码 |
| VARCHAR(n) | 可变长度 | 姓名、地址 |
| TEXT | 长文本 | 文章内容 |
| LONGTEXT | 超长文本 | 大型文档 |

### 5.3 日期时间类型

| 类型 | 精度 | 适用场景 |
|------|------|----------|
| DATE | 日期 | 生日、日期 |
| TIME | 时间 | 时刻 |
| DATETIME | 日期+时间 | 记录时间 |
| TIMESTAMP | 时间戳 | 更新时间 |

## 六、索引设计

### 6.1 索引类型

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| 普通索引 | 最基本的索引 | 一般查询 |
| 唯一索引 | 值唯一 | 用户名、邮箱 |
| 主键索引 | 唯一且非空 | 主键 |
| 复合索引 | 多个字段组合 | 联合查询 |
| 全文索引 | 全文搜索 | 文章内容 |

### 6.2 创建索引

```sql
-- 创建普通索引
CREATE INDEX idx_user_name ON users(name);

-- 创建唯一索引
CREATE UNIQUE INDEX idx_user_email ON users(email);

-- 创建复合索引
CREATE INDEX idx_order_user_status ON orders(user_id, status);
```

### 6.3 索引使用原则

- **优点**：加速查询
- **缺点**：减慢写入速度，占用空间

**使用建议：**
- 经常用于 WHERE 条件的列加索引
- 经常用于 JOIN 的列加索引
- 不要过度索引

## 七、命名规范

### 7.1 表命名

```sql
-- 表名：小写，下划线分隔，复数形式
users, orders, products, order_items

-- 中间表：用下划线连接两个表名
user_roles, product_categories
```

### 7.2 字段命名

```sql
-- 字段名：小写，下划线分隔
user_id, order_date, total_amount

-- 时间字段
created_at, updated_at, deleted_at

-- 状态字段
status, is_active, has_deleted
```

### 7.3 约束命名

```sql
-- 主键约束
pk_users_id

-- 外键约束
fk_orders_user_id

-- 唯一约束
uk_users_email
```

## 十、常见错误

### 10.1 反模式

```sql
-- 错误：将多个值存储在一个字段中
CREATE TABLE users (
  id INT PRIMARY KEY,
  hobbies VARCHAR(200) -- "篮球,足球,游泳"
);

-- 正确：使用关联表
CREATE TABLE user_hobbies (
  user_id INT,
  hobby_id INT,
  PRIMARY KEY (user_id, hobby_id)
);
```

### 10.2 过度设计

- 创建不必要的表
- 添加过多的索引
- 过度规范化导致表关联复杂

### 10.3 缺乏文档

- 没有注释说明字段用途
- 没有 ER 图
- 没有数据字典

## 总结

数据库设计是开发的基础，好的设计：

1. **清晰的实体关系**：明确表之间的关联
2. **合理的范式**：1NF-3NF 平衡
3. **恰当的索引**：加速查询但不影响写入
4. **统一的命名**：易于理解和维护
5. **完整的文档**：便于团队协作
