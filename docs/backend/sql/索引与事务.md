# 索引与事务

## 一、索引原理

### 1.1 什么是索引

索引是一种数据结构，用于加速数据库查询。它就像一本书的目录，可以快速定位到需要的数据。

### 1.2 B+树索引

大多数数据库使用 B+树作为索引结构：

```
          根节点
            |
    ┌──────┴──────┐
    │             │
  中间节点      中间节点
    │             │
 ┌──┴──┐      ┌──┴──┐
 │     │      │     │
叶子节点   叶子节点   叶子节点   叶子节点
```

**B+树特点：**
- 平衡树，所有叶子节点在同一层
- 叶子节点形成链表，方便范围查询
- 非叶子节点不存储数据，只存储索引键

### 1.3 索引类型对比

| 索引类型 | 结构 | 优点 | 缺点 |
|----------|------|------|------|
| B+树 | 树形结构 | 范围查询快 | 写入慢 |
| 哈希索引 | 哈希表 | 等值查询快 | 不支持范围查询 |
| 全文索引 | 倒排索引 | 全文搜索 | 占用空间大 |

## 二、索引类型详解

### 2.1 主键索引

```sql
-- 创建主键索引
CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT
);
```

**特点**：
- 唯一且非空
- 自动创建
- 加速主键查询

### 2.2 唯一索引

```sql
-- 创建唯一索引
CREATE UNIQUE INDEX idx_user_email ON users(email);
```

**特点**：
- 值必须唯一
- 可包含空值（MySQL）
- 防止重复数据

### 2.3 普通索引

```sql
-- 创建普通索引
CREATE INDEX idx_user_name ON users(name);
```

**特点**：
- 最常用的索引类型
- 可包含重复值
- 加速查询

### 2.4 复合索引

```sql
-- 创建复合索引
CREATE INDEX idx_order_user_status ON orders(user_id, status);
```

**最左前缀原则**：
- `WHERE user_id = 1` ✓ 使用索引
- `WHERE user_id = 1 AND status = 'paid'` ✓ 使用索引
- `WHERE status = 'paid'` ✗ 不使用索引

### 2.5 覆盖索引

```sql
-- 覆盖索引：索引包含查询所需的所有列
CREATE INDEX idx_user_name_email ON users(name, email);

-- 这个查询可以只使用索引完成（覆盖索引）
SELECT name, email FROM users WHERE name = '张三';
```

## 三、索引优化策略

### 3.1 何时创建索引

```sql
-- 经常用于 WHERE 条件
CREATE INDEX idx_order_status ON orders(status);

-- 经常用于 JOIN
CREATE INDEX idx_order_user_id ON orders(user_id);

-- 经常用于 ORDER BY
CREATE INDEX idx_user_created_at ON users(created_at);
```

### 3.2 何时不创建索引

- 表数据量小
- 经常更新的列
- 低选择性的列（如性别）

### 3.3 索引维护成本

```sql
-- 索引会减慢写入操作
INSERT INTO users (name, email) VALUES ('张三', 'test@example.com');
-- 需要更新所有相关索引
```

### 3.4 EXPLAIN 分析

```sql
-- 使用 EXPLAIN 分析查询
EXPLAIN SELECT * FROM users WHERE name = '张三';
```

**EXPLAIN 输出字段：**
- `type`：访问类型（ALL, index, range, ref, eq_ref）
- `key`：使用的索引
- `rows`：扫描的行数
- `Extra`：额外信息

## 四、事务基础

### 4.1 什么是事务

事务是一组原子性的数据库操作，要么全部成功，要么全部失败。

### 4.2 ACID 特性

| 特性 | 说明 | 示例 |
|------|------|------|
| **Atomicity** | 原子性 | 转账要么成功要么回滚 |
| **Consistency** | 一致性 | 转账前后总金额不变 |
| **Isolation** | 隔离性 | 事务之间互不干扰 |
| **Durability** | 持久性 | 提交后数据永久保存 |

### 4.3 事务控制

```sql
-- 开始事务
START TRANSACTION;

-- 执行操作
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

-- 提交事务
COMMIT;

-- 回滚事务
ROLLBACK;
```

## 五、事务隔离级别

### 5.1 隔离级别定义

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|----------|------|------------|------|
| READ UNCOMMITTED | ✓ | ✓ | ✓ |
| READ COMMITTED | ✗ | ✓ | ✓ |
| REPEATABLE READ | ✗ | ✗ | ✓ |
| SERIALIZABLE | ✗ | ✗ | ✗ |

### 5.2 设置隔离级别

```sql
-- 设置会话隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 设置全局隔离级别
SET GLOBAL TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### 5.3 各隔离级别特点

**READ UNCOMMITTED（读未提交）**：
- 最低级别
- 可能读取未提交的数据

**READ COMMITTED（读已提交）**：
- 只能读取已提交的数据
- Oracle 默认级别

**REPEATABLE READ（可重复读）**：
- 同一事务内多次读取相同数据结果一致
- MySQL InnoDB 默认级别

**SERIALIZABLE（串行化）**：
- 最高级别
- 事务串行执行
- 性能最低

## 六、锁机制

### 6.1 锁的类型

| 锁类型 | 说明 | 粒度 |
|--------|------|------|
| 表锁 | 锁定整个表 | 粗粒度 |
| 行锁 | 锁定单行数据 | 细粒度 |
| 页锁 | 锁定数据页 | 中等粒度 |

### 6.2 行锁示例

```sql
-- 事务1
START TRANSACTION;
SELECT * FROM products WHERE id = 1 FOR UPDATE; -- 加锁

-- 事务2（等待锁释放）
START TRANSACTION;
UPDATE products SET stock = stock - 1 WHERE id = 1; -- 阻塞
```

### 6.3 乐观锁 vs 悲观锁

```sql
-- 悲观锁：先加锁再操作
SELECT * FROM products WHERE id = 1 FOR UPDATE;
UPDATE products SET stock = stock - 1 WHERE id = 1;

-- 乐观锁：先读取版本号，更新时检查
UPDATE products 
SET stock = stock - 1, version = version + 1 
WHERE id = 1 AND version = 5;
```

## 七、死锁

### 7.1 死锁示例

```sql
-- 事务1
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2; -- 等待

-- 事务2
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE id = 2;
UPDATE accounts SET balance = balance + 100 WHERE id = 1; -- 等待
```

### 7.2 死锁预防

- 按固定顺序访问资源
- 保持事务简短
- 使用较低的隔离级别
- 设置事务超时

## 八、实战案例

### 8.1 案例一：优化慢查询

```sql
-- 慢查询
SELECT * FROM orders WHERE user_id = 123; -- 全表扫描

-- 添加索引
CREATE INDEX idx_order_user_id ON orders(user_id);

-- 再次查询（使用索引）
SELECT * FROM orders WHERE user_id = 123;
```

### 8.2 案例二：转账事务

```sql
START TRANSACTION;

-- 检查余额
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;

-- 扣款
UPDATE accounts SET balance = balance - 100 WHERE id = 1;

-- 检查余额是否足够
SELECT balance FROM accounts WHERE id = 1;
IF balance < 0 THEN ROLLBACK;

-- 收款
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

COMMIT;
```

### 8.3 案例三：库存扣减（乐观锁）

```sql
-- 读取当前库存和版本
SELECT stock, version FROM products WHERE id = 1;

-- 更新库存（乐观锁）
UPDATE products 
SET stock = stock - 1, version = version + 1 
WHERE id = 1 AND version = current_version AND stock >= 1;
```

## 九、性能最佳实践

### 9.1 索引最佳实践

1. **只为必要的列创建索引**
2. **使用复合索引时遵循最左前缀原则**
3. **定期重建或优化索引**
4. **避免在索引列上进行函数操作**

### 9.2 事务最佳实践

1. **保持事务简短**
2. **避免在事务中进行耗时操作**
3. **使用合适的隔离级别**
4. **处理死锁异常**

### 9.3 常见陷阱

```sql
-- 避免：在索引列上使用函数
SELECT * FROM users WHERE YEAR(created_at) = 2024; -- 不使用索引

-- 推荐：重写为范围查询
SELECT * FROM users WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'; -- 使用索引

-- 避免：SELECT *
SELECT * FROM orders WHERE user_id = 1; -- 查询所有列

-- 推荐：只查询需要的列
SELECT id, order_date, total_amount FROM orders WHERE user_id = 1; -- 可能使用覆盖索引
```

## 总结

索引和事务是数据库的核心概念：

1. **索引**：加速查询，理解 B+树结构，遵循最左前缀原则
2. **事务**：保证数据一致性，理解 ACID 特性
3. **隔离级别**：根据业务需求选择合适的隔离级别
4. **锁机制**：理解行锁和表锁，避免死锁
5. **性能优化**：使用 EXPLAIN 分析查询，合理创建索引
